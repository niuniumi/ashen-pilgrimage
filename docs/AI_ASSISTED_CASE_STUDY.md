# 《灰烬圣途》AI 辅助开发案例

## 项目定位

目标不是制作一次性网页演示，而是把已有原型整理为可持续迭代、可完整通关、可上线发布的浏览器游戏产品。

## 人与 AI 的分工

项目作者负责产品目标、玩法取舍、视觉与听觉反馈、问题判断和最终验收。OpenAI Codex 作为开发协作者，负责代码审计、架构调整、功能实现、内容扩充、自动化测试、素材授权核验、构建与部署。

这一定义避免把 AI 生成等同于完整工程所有权，也保留了作者在产品决策中的核心作用。

## 关键工程工作

1. 将局内随机统一为可序列化种子和随机游标，使地图、遭遇、洗牌、奖励、宝箱与事件能够复现。
2. 建立 v4 存档、场景级恢复路由和战斗检查点，处理旧存档迁移、异常节点回滚和重复结算。
3. 为三名角色建立独立资源系统，并将首领阶段、敌人职能和遭遇节奏拆分为系统模块。
4. 将每章路线扩充到 12 层，平衡战斗、精英、商店、休息、宝箱和事件密度。
5. 重建音频路由：8 条独立 BGM、50 个效果音样本、响度标准化、音效轮换和场景淡入淡出。
6. 重建像素表现层：六张场景背景、16 个高精度角色资产、本地中文像素字体、像素 UI 与战斗反馈。
7. 增强战斗反馈：出牌轨迹、打击停顿、冲击爆点、镜头震动、首领阶段爆发和结果页演出。
8. 建立单元测试、确定性战斗模拟、内容校验、角色矩阵、存档恢复和发布流程检查。

## 产品规模

- 3 名角色与 3 套资源循环
- 70 张卡牌
- 24 件正式遗物
- 28 种敌人，其中 3 名三阶段首领
- 21 个事件与 9 条角色结局
- 三章共 36 层路线

## 质量标准

- 所有地图节点可达，每个节点最多两条去向。
- 相同种子产生相同局内序列，恢复存档后继续保持一致。
- 所有正式内容通过数据结构校验。
- 生产音频通过格式、采样率、时长和响度检查。
- 发布前执行 70 项单元测试、180 场战斗模拟、浏览器流程回归和生产构建。

## 结果

项目以 GitHub Pages 生产站点交付，并通过公开 GitHub 仓库展示源代码、测试体系、素材来源和 AI 辅助开发方法。每次主分支发布均由 GitHub Actions 重新执行核心测试与资产审计。
