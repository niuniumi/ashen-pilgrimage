# 《灰烬圣途》发布级终局优化设计

## 目标

在不缩减关卡、不破坏现有存档、不重写已稳定战斗规则的前提下，完成一次可上线、可量化、可回归的终局优化。重点解决首屏加载过重、结果页风格断裂、非战斗场景反馈偏平、资源加载缺少边界和发布验收不足的问题。

## 成功标准

- 初次进入主菜单不再下载全章节图片与音频；首屏编码传输量控制在 6 MB 内，初始资源请求控制在 24 个内。
- 主菜单可交互前仅加载菜单所需资源；第二、三章背景、角色和 BGM 不得出现在首屏资源记录中。
- 各场景只加载自身及当前章节所需资源，切换时有统一的像素风加载反馈；加载失败时给出明确错误并允许重试或安全返回。
- 胜利与失败结果页拥有不同构图、清晰统计层级、完整按钮状态；失败墓碑必须为透明底、纯像素风、与全局美术一致的正式资产。
- 奖励、事件、休息、地图等场景具有明确选择状态、确认反馈和短促过渡动效，且关闭动画设置后不影响操作。
- 现有三章流程、Boss 解锁、敌人朝向、存档迁移、角色身份、牌组与战斗结算全部通过自动化回归。
- 构建无错误，桌面与窄屏截图无重叠、裁切、空白画布或资源缺失；GitHub Pages 线上版本通过独立冒烟测试。

## 范围拆分

### 子项目一：资源加载与性能

建立集中式资源包目录，将资源按 `boot`、`menu`、`character-select`、`map-act-N`、`battle-act-N`、`boss-act-N`、`folio`、`result` 分类。`PreloadScene` 只加载启动包；每个场景在自己的 `preload()` 中声明依赖。Phaser 全局缓存负责跨场景复用，重复进入场景不得重复下载。

移除主菜单进入 180 毫秒后加载全部剩余音频和视觉资源的行为。BGM 与 SFX 按使用场景拆包：通用 UI 声音可进入启动包，战斗、地图、Boss、结果音频随对应场景加载。资源包加载必须幂等，并提供统一进度视图、错误收集与重试入口。

运行时美术使用无损 WebP，PNG 作为源资产保留在不参与 Vite 发布的源目录。转换必须保持尺寸和透明通道，不对像素画执行有损压缩或平滑缩放。生产清单只引用运行时文件，资产 QA 校验不存在双份发布、错误扩展名或缺失键。

### 子项目二：结果页与正式像素资产

结果页采用左右不对称的终章构图：左侧为角色或墓碑主视觉，右侧为结局叙事、关键统计和牌组摘要，底部为明确的下一步操作。胜利使用余烬金、象牙白和冷铁色；失败使用灰烬黑、暗血红和冷月灰，不使用绿色墓地背景。

失败墓碑由图像生成工具产出正式透明 PNG，再经过像素边缘、透明通道、尺寸和运行时绑定检查。墓碑轮廓应正面略偏三分之二视角，带破损圣徽、熄灭烛泪和少量灰烬，不出现写实照片质感、绿色草地、复杂背景或文字乱码。

统计区优先展示章节进度、击败敌人、用时、遗物与誓约、最终金币；牌组摘要使用类型与升级状态分组，避免长文本堆叠。结果页只记录一次统计，重复刷新或返回不得重复增加胜负次数。

### 子项目三：非战斗场景反馈

奖励、事件和休息场景统一采用“浏览选择 -> 明确选中 -> 确认结算”的反馈链路。可交互项有稳定尺寸、键盘焦点、悬停、按下、已选和禁用状态；确认后立即锁定重复输入，再播放 150-300 毫秒的状态过渡。

地图节点保持当前三章内容与拓扑，不缩减路线。节点类型图标继续保持差异化，当前节点、可达节点、已完成节点和不可达节点在颜色、亮度与轮廓上同时区分。线路层级低于节点，装饰线不得穿过文字或掩盖路径关系。

动效只承担状态说明：卡牌奖励轻微抬升与聚焦，事件选择使用短促墨迹/火星确认，休息选项使用暖光或铁砧火星，地图解锁使用沿路径传播的余烬。所有循环环境动画必须可停用，并遵守现有动画设置。

### 子项目四：质量门槛与上线

新增资源预算 QA，使用 Playwright 的 Performance API 记录首屏资源数、`encodedBodySize` 和资源键，断言不存在跨章节预载。新增场景资源 QA，验证每个场景所需纹理和音频在 `create()` 前已进入缓存，且加载失败不会卡死在空白页面。

扩展结果页、奖励、事件、休息与地图的截图回归。截图覆盖 1536x864、1280x720 和窄屏比例；检查空白画布、文字溢出、按钮遮挡、透明资产黑底、敌我朝向与 UI 漂移。

最终发布顺序固定为：单元测试 -> 内容与资产 QA -> 战斗模拟 -> 三章完整流程 -> 桌面/窄屏视觉回归 -> 生产构建 -> 本地生产预览 -> 推送 -> GitHub Actions -> 线上冒烟。任何一步失败都停止发布并修复，不以旧截图或本地开发服务器结果代替线上验证。

## 架构边界

- `AssetBundleCatalog` 只描述资源，不直接控制场景。
- `SceneAssetLoader` 负责幂等排队、进度、失败状态和缓存检查，不包含业务逻辑。
- 各场景声明自身资源包，不访问其他章节内部资源。
- `AudioManager` 继续负责播放、音量、解锁和淡入淡出；资源加载职责从中分离。
- 结果页布局拆为主视觉、叙事、统计、牌组摘要和操作区，避免继续膨胀单一渲染函数。
- 现有战斗系统、地图生成规则和存档结构只增加兼容测试，不做无关重写。

## 数据流

1. 场景进入 `preload()`，根据运行状态解析资源包键。
2. 加载器过滤 Phaser 缓存中已有资源，排队缺失项并更新统一进度视图。
3. 加载完成后进入 `create()`；若关键资源失败，显示场景内错误面板，允许重试或返回主菜单。
4. 场景业务只使用稳定资源键，不拼接任意路径。
5. 性能 QA 从浏览器资源记录反向校验实际网络请求与资源包声明一致。

## 视觉与动效规范

- 保持现有冷铁、余烬金、暗血红、骨白的像素哥特体系，不引入绿色墓地、写实剪贴图或通用渐变卡片。
- 像素资产使用整数缩放和最近邻采样；不使用平滑滤镜。
- 微交互 80-120 毫秒，选中与确认 160-260 毫秒，场景编排 300-550 毫秒，总序列不超过 700 毫秒。
- 元素进入使用 ease-out，退出使用 ease-in，路径解锁使用 ease-in-out；连续动画仅用于火焰、灰烬和雾。
- 关闭动画时，所有状态变化必须即时完成且仍可辨识。

## 兼容与风险控制

- 不修改存档键名、角色 ID、敌人 ID、章节编号和卡牌 ID。
- 保留 PNG 源文件但避免与 WebP 同时进入生产包；转换脚本必须可重复执行。
- 音频自动播放仍受浏览器策略约束；首次指针输入后解锁，未解锁时不阻塞场景。
- 资源加载失败不得进入永久 `loading` 状态；重试前清理失败监听和加载器所有权。
- 场景切换期间取消属于旧场景的延时任务，避免关场后继续创建 VFX 或触发音频。

## 非目标

- 不缩减或新增章节、节点、敌人和卡牌内容。
- 不重新设计伤害、能量、抽牌、敌人意图和 Boss 数值体系。
- 不变更存档格式或清空玩家现有进度。
- 不引入新的运行时框架或重量级 UI 依赖。
